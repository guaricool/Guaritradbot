🎁 Lo que podemos tomar de kalshi-ai (post-revisión 2026-07-11)

🆕 LLM cost-cap pattern

daily_ai_cost_limit = 10.0 chequeado ANTES de cada API call. Cuando se gasta, no trade hasta el otro día. Único en el benchmark con este guardrail. Aplicable si añadimos LLM al debate (Sprint 47+).

Effort: S (½ día) · Impact: medio — solo si adoptamos LLM

🆕 Multi-criteria exit (track.py)

5 criterios en cascada: market-resolution → stop-loss → take-profit → time-based (max_hold_hours) → emergency para posiciones sin SL. Nosotros tenemos SL/TP/OCO; falta time-based exit (cerrar posición vieja sin movimiento en N días).

Effort: S (1 día, copiar patrón) · Impact: medio — reduce tied-up capital

🆕 Cash reserves manager (17KB)

0.5% mínimo cash reserve con auto-halt. No lo tenemos — nuestro gate es per-position y per-day, no portfolio-level cash buffer. En cuenta de $20 no aplica; en $1000+ sí.

Effort: M (1 sprint) · Impact: bajo ahora, alto si escalamos balance

🆕 Multi-model fallback chain

OpenRouter con primary model + 2-3 fallbacks on error + persistent cost tracker. Patrón transferible. Ellos usan Claude Sonnet 4.5 primary; nosotros hoy no usamos LLM pero si lo adoptamos este es el patrón.

Effort: S (cuando adoptemos LLM) · Impact: alto para resiliencia

🆕 Graceful SIGINT (signal-type logging)

Ellos loggean SIGINT/SIGTERM/SIGHUP y terminan el ciclo in-flight. Nuestro BOT_STOP_KEYBOARDINT (recién agregado en Sprint 46N follow-up) es parcial: el scheduler loop no verifica entre ciclos. Vale la pena cerrar el loop.

Effort: S (½ día) · Impact: medio — evita cierres corruptos en Ctrl+C

🆕 CLI unificada (cli.py, 30KB)

python cli.py run|dashboard|status|health|close-all|scores|history. Tenemos API pero no CLI. Operacionalmente útil para debugging y para correr una sola estrategia sin el daemon completo.

Effort: M (2-3 días) · Impact: bajo (API cubre el 80%)

🆕 Category scorer 0-100

Score 0-100 por categoría de mercado (gatea position size). No es lo mismo que nuestro allocation policy (40/40/10/10 estático), pero podríamos agregar un "symbol score" dinámico (ej. BTC en tendencia = 85 → max 20%; BTC choppy = 40 → max 5%).

Effort: M (1 sprint) · Impact: medio — sizing más inteligente

🆕 WebSocket market data (kalshi_ws.py)

WebSocket nativo para live prices + order book. Nosotros poleamos yfinance/ccxt cada 2 min (audit A7). ccxt soporta WebSocket para binance; el cambio baja latencia y arregla el gap de 2 min en gap-downs.

Effort: M-L (1-2 sprints, audit A7) · Impact: alto — fixes A7
🚧 Brechas reales (post-Sprint 46N)

❌ ML/AI real (sigue siendo gap)

Tenemos GP + alpha zoo pero NO label/train/predict classical ML. intelligent-trading-bot usa talib/tsfresh/tbstats → modelos supervisados. kalshi-ai usa LLM directo. Quick win: Sprint 47 con talib + XGBoost baseline (label=forward 4h return). Esto también desbloquea el debate LLM con signal hibrido (reglas + ML).

Effort: M-L (1-2 sprints) · Impact: muy alto (cierra el último gap grande)

⚠️ Indicadores vs talib

57+ indicadores custom pero talib = 100+. Quick win: pip install talib-binary (compatible binance VPS) + integrar 30-40 indicators más. 1 día.

Effort: S (1 día) · Impact: medio (subimos ⚠️→✅✅)

⚠️ Hyperopt maturity (Optuna)

Tenemos HyperoptManager (grid) + GP. freqtrade tiene Optuna + Bayesian + multi-fidelity. Quick win: pip install optuna y migrar grid → TPE. Cierra el gap de eficiencia en hyperopt.

Effort: S (2-3 días) · Impact: bajo-medio

⚠️ A8 audit: audit.jsonl sin rotación

Sin rotación en ningún lado. EQUITY_UPDATE cada 2 min (~720/día). _audit_tail_loop re-parsea archivo entero cada 1s. Costo lineal creciente. Quick win: rotación mensual + tail con offset de bytes + sacar EQUITY_UPDATE del ledger (ya está en equity_state.json).

Effort: M (1 sprint) · Impact: medio — performance + log management

⚠️ A7 audit: SL/TP con precios Yahoo vs broker

BTC-USD de Yahoo es un índice, no el libro de binance.us. Un stop cruzado en Yahoo dispara un market order en libro que puede estar lejos. Fix: usar ccxt fetch_ticker (cripto) + Alpaca last trade (equity) para SL/TP comparison. yfinance solo para históricos/indicadores.

Effort: M · Impact: alto — fixes audit A7

⚠️ A5 audit: scheduler mono-hilo

El ciclo horario y fast_monitor_tick corren secuencialmente en 1 hilo. Ciclo lento suspende SL/TP. Fix: fast tick en hilo propio + cachear correlación/CVaR por ciclo. También arregla A6 (fast tick blind alert).

Effort: M · Impact: medio

🧹 M7: borrar dashboard.py (Streamlit muerto)

163KB / 4.043 líneas de código muerto. Libera streamlit, streamlit-autorefresh, plotly de requirements (~150-200MB de imagen). Mata 16 errores de test_dashboard_fmt y deja 1 test stale (Dockerfile user).Es housekeeping puro, no suma score.
